Laboratorio 4: Enrutamiento Estático y Gateway de Última Esperanza (ISP)

Este escenario introduce el concept de demarcación perimetral y transporte WAN. Modificaremos la arquitectura anterior desconectando el Servidor DNS del router local y colocándolo detrás de un nuevo Router de Borde que simulará al Proveedor de Servicios de Internet (ISP). Aplicaremos rutas estáticas específicas y de última esperanza para forzar la convergencia global de la red.

📌 El Secreto del Enrutamiento Bidireccional: En redes, el enrutamiento siempre es de ida y vuelta. Que el Router de tu empresa (`R1`) sepa cómo mandar un paquete a Internet no sirve de nada si el Router del ISP no tiene una ruta de regreso para saber en qué interfaz interna de la WAN se aloja tu rango de IPs corporativas.

Fase 1: Reestructuración de la Topología Física y Enlaces

Desconectaremos el Servidor DNS del puerto de R1 para ubicarlo en su verdadero destino jerárquico: la red del ISP.

Topología Nueva Red de Transporte WAN

Modificaciones en el Lienzo de Packet Tracer:

  • Router Corporativo (R1): Mantiene sus tres conexiones locales hacia los switches (VLAN 10, 20 y 30).
  • Nuevo Router Borde (ISP): Modelo ISR 4331. Hará de pasarela hacia la nube.
  • Server-DNS: Se desconecta de R1 y se conecta **directamente al puerto Gi0/0/1 del Router ISP**.

Esquema de Cableado e Interconexiones

[Switches S1, S2, S3] ─────> Conectados a R1 (Mismo esquema Lab 3)

[Router R1] (Gi0/0/3) ─── Link WAN Punto a Punto ─── (Gi0/0/0) [Router ISP]
                          Red: 10.0.0.0 /30

[Router ISP] (Gi0/0/1) ──────────────────────────────> [Server-DNS] (Fa0)
                          Red LAN Pública: 8.0.0.0 /8

Dispositivo / Interfaz Dirección IP Máscara de Red Descripción / Propósito

Fase 3: ¿Por qué NO usamos enlaces Trunk en esta práctica?

Una confusión habitual al diseñar topologías es asumir que siempre que existan VLANs debe haber enlaces troncales (Trunks) hacia el router. En este laboratorio hemos optado por un enfoque de Interfaces Físicas Dedicadas (también conocido como modelo tradicional MIE).

Análisis Lógico El porqué de los Puertos en Modo Acceso

En el diseño actual, cada Switch independiente (S1, S2, S3) tiene un cable directo que va a parar a una boca física exclusiva del Router R1 (Gi0/0/0, Gi0/0/1 y Gi0/0/2 respectivamente).

  • Sin Etiquetas (Untagged): Como por cada cable viaja el tráfico de una sola y única subred, los paquetes no necesitan ser marcados con el estándar IEEE 802.1Q. Las tramas Ethernet se desplazan de forma limpia y plana.
  • Configuración de los Switches: Los puertos de los switches que se conectan hacia el router se configuran exactamente igual que si fueran una PC: en modo acceso (switchport mode access) asignados a su respectiva VLAN.

Comparativa de Arquitectura para el Próximo Paso

Este laboratorio sirve como la antesala perfecta para entender la necesidad técnica de los enlaces troncales. Evaluemos la diferencia de ingeniería:

Criterio Técnico Modelo Actual (Interfaces Dedicadas) Próximo Nivel (Router-on-a-Stick / Trunk)
Cables físicos hacia el Router Muchos: Uno por cada switch o VLAN de la empresa. Uno solo: Un único enlace físico centralizado (Troncal).
Estado del puerto del Switch al Router switchport mode access (Pertenecen a una sola VLAN). switchport mode trunk (Transporta múltiples VLANs concurrentes).
Configuración en el Router Direcciones IP asignadas directo en puertos físicos (interface Gi0/0/0). Subinterfaces lógicas virtuales con tag de software (interface Gi0/0/0.10).
Consumo de Hardware Ineficiente: Si la empresa tiene 10 VLANs, necesitás un router carísimo con 10 bocas físicas. Altamente Eficiente: Soportás decenas de VLANs usando un único puerto físico del router.
🚀 Nota de Progresión Pedagógica: Mantener este diseño nos permite aislar el tráfico por hardware (cableado físico). En la próxima práctica, unificaremos los switches y bajaremos un único cable hacia el router. Allí verás cómo el comando switchport mode trunk se vuelve obligatorio para que el router pueda discriminar qué paquete pertenece a qué departamento mediante software.

Fase 4: Scripts de Configuración de la CLI y Rutas Estáticas

Aplicá las siguientes plantillas en los respectivos terminales lógicos. Notarás que el comando ip route es la pieza fundamental de esta sección.

CLI Router R1: Interfaces WAN y Ruta por Defecto
R1> enable
R1# configure terminal

# Configurar el enlace WAN punto a punto hacia el ISP
R1(config)# interface gigabitEthernet 0/0/3
R1(config-if)# description Enlace_Hacia_ISP
R1(config-if)# ip address 10.0.0.1 255.255.255.252
R1(config-if)# no shutdown
R1(config-if)# exit

# --- ENRUTAMIENTO ESTÁTICO DE SALIDA (Ruta por Defecto) ---
R1(config)# ip route 0.0.0.0 0.0.0.0 10.0.0.2
R1(config)# do write memory
CLI Router ISP: Puertos WAN/LAN y Rutas de Retorno
Router> enable
Router# configure terminal
Router(config)# hostname ISP

# Interfaz WAN hacia R1
ISP(config)# interface gigabitEthernet 0/0/0
ISP(config-if)# description Enlace_Hacia_R1_Corp
ISP(config-if)# ip address 10.0.0.2 255.255.255.252
ISP(config-if)# no shutdown
ISP(config-if)# exit

# Interfaz LAN hacia el Servidor DNS
ISP(config)# interface gigabitEthernet 0/0/1
ISP(config-if)# description Red_Servidores_Publicos
ISP(config-if)# ip address 8.0.0.1 255.0.0.0
ISP(config-if)# no shutdown
ISP(config-if)# exit

# --- ENRUTAMIENTO ESTÁTICO DE RETORNO HACIA LA EMPRESA ---
ISP(config)# ip route 192.168.10.0 255.255.255.0 10.0.0.1
ISP(config)# ip route 192.168.20.0 255.255.255.0 10.0.0.1
ISP(config)# ip route 192.168.30.0 255.255.255.0 10.0.0.1
ISP(config)# do write memory

Fase 5: Verificación de Tablas y Pruebas del Plano de Datos

Con las rutas inyectadas, auditaremos la tabla de enrutamiento para asegurar el prefijo "S" y ejecutaremos las comprobaciones de conectividad.

Troubleshooting Auditoría de Comandos y Diagnóstico de Rutas

1. Inspección de Rutas en R1

Ejecutá show ip route en R1. Al final de la tabla verás la siguiente línea destacada:

S* 0.0.0.0/0 [1/0] via 10.0.0.2

Nota de examen: La letra S significa Estática y el asterisco (*) indica que es una candidata a ruta por defecto (Gateway of Last Resort).

2. Inspección de Rutas en ISP

Ejecutá show ip route static en el Router ISP para comprobar que solo liste los caminos manuales ingresados:

ISP# show ip route static
S     192.168.10.0/24 [1/0] via 10.0.0.1
S     192.168.20.0/24 [1/0] via 10.0.0.1
S     192.168.30.0/24 [1/0] via 10.0.0.1

3. Prueba de Trazabilidad Extremo a Extremo (PC-V10 a Server-DNS)

Abrí el Command Prompt en la PC-V10 y ejecutá una traza hacia la IP pública del DNS para verificar los saltos institucionales:

C:\> tracert 8.8.8.8

Análisis del Resultado Exitoso de la Traza:

  1. Salto 1 (192.168.10.1): El paquete llega a su Gateway local (Interfaz Gi0/0/0 de R1).
  2. Salto 2 (10.0.0.2): R1 no conoce la IP 8.8.8.8, pero usa su ruta por defecto y se la entrega al ISP a través del enlace WAN.
  3. Salto 3 (8.8.8.8): El ISP recibe el paquete, revisa sus redes directamente conectadas en la Gi0/0/1 y entrega exitosamente el paquete al Servidor DNS.
