01 — Fundamentos
¿Qué es ContainerLab?

ContainerLab es una herramienta de código abierto que permite crear laboratorios de redes virtuales usando contenedores Docker. Cada nodo (router, switch, PC, servidor) es un contenedor ligero que se interconecta mediante interfaces virtuales, simulando cables de red reales.

🐳

Basado en Docker

Cada dispositivo de red es un contenedor. Arrancan en segundos y consumen pocos recursos.

📄

Definición en YAML

La topología completa se describe en un archivo .yml — nodos, enlaces, imágenes y configuraciones. YML (YAML) es un formato de texto para definir configuraciones y datos mediante indentación.

⚡

Arranque rápido

Un laboratorio completo con routers, switches y servidores puede estar listo en menos de 1 minuto.

🔬

Software real

Se ejecuta software de red real (FRRouting, OVS, Linux) — no simuladores, sino implementaciones completas.

📌 Importante: En esta práctica, el laboratorio ya está desplegado por el docente. Tu tarea es conectarte a los dispositivos mediante Guacamole y explorar la red funcionando.

03 — Componentes
Open vSwitch (OVS) — El switch virtual

Open vSwitch es un switch de red implementado por software, compatible con el estándar IEEE 802.1Q. En este laboratorio actúa como switch de distribución: separa el tráfico en VLANs usando etiquetas en los frames Ethernet.

🏷️

Puertos de acceso

Los puertos conectados a las PCs tienen un tag=N — todo el tráfico que entra se etiqueta con el número de VLAN correspondiente.

🔀

Puerto trunk

El enlace hacia el router R1 lleva tráfico de múltiples VLANs simultáneamente, identificadas por sus etiquetas 802.1Q.

🔒

Aislamiento

PCs en distintas VLANs no pueden comunicarse directamente — el tráfico debe pasar por el router R1 para cruzar entre segmentos.

🐧

Software libre

OVS es open source y se usa en producción en plataformas cloud como OpenStack y VMware NSX.

04 — Acceso remoto
Apache Guacamole — Tu terminal en el navegador

Guacamole es un gateway de escritorio remoto sin cliente — todo funciona desde el navegador web, sin instalar software adicional. En este laboratorio proporciona acceso SSH a cada contenedor.

🌐

Sin cliente

Solo necesitás un navegador. No se instala nada en tu computadora.

🔐

Acceso SSH

Cada nodo del lab tiene una conexión SSH pre-configurada en Guacamole.

📋

Portapapeles

Para pegar comandos usá el panel lateral: Ctrl+Alt+Shift para abrirlo.

🔑 Cómo conectarse a Guacamole
  1. Abrí tu navegador y accedé a la URL que indica el docente (ej: https://lab.trigoproject404.website/#/)
  2. Ingresá con usuario estudiante y la contraseña indicada en clase
  3. En el panel de conexiones verás los nodos disponibles: PC1, PC2, PC3, OVS-SW, R1, Web-Server
  4. Hacé clic en el nodo que quieras acceder — se abre una terminal SSH en el navegador
  5. Para pegar comandos: Ctrl+Alt+Shift → pestaña Portapapeles → pegá el texto → cerrá el panel
  6. Para abrir dos sesiones simultáneas, abrí una segunda pestaña del navegador y volvé a entrar a Guacamole

A2 Prueba de conectividad entre VLANs 10 pts
🖥 Guacamole — Sesión requerida

Conectate a PC1 desde Guacamole.

Probá la conectividad desde PC1 hacia los demás dispositivos:

Desde PC1
ping -c 3 192.168.10.1    # Gateway VLAN 10 (R1)

ping -c 3 192.168.30.10   # PC3 — distinta VLAN

💡 Para analizar

¿El tráfico entre PC1 y PC2 pasa por el router R1? ¿Por qué dos PCs en distintas VLANs no pueden comunicarse directamente aunque estén en el mismo switch físico?

A3 Inspección del router R1 10 pts
🖥 Guacamole — Sesión requerida

Conectate a R1 desde Guacamole.

Desde R1
ip addr show              # Ver subinterfaces eth5.10/.20/.30
ip route show             # Ver tabla de enrutamiento
vtysh -c "show ip route"  # Tabla de rutas FRR (formato Cisco)

En tu informe: copiá la salida de ip route show y explicá para qué sirve cada ruta que ves.

A4 Inspección del switch OVS 10 pts
🖥 Guacamole — Sesión requerida

Conectate a OVS-SW desde Guacamole.

Desde OVS-SW
ovs-vsctl show                    # Ver configuración del bridge br0
ovs-vsctl list-ports br0          # Listar todos los puertos
ovs-vsctl get port eth1 tag        # Ver VLAN tag del puerto de PC1
ovs-vsctl get port eth5 trunks     # Ver VLANs del trunk hacia R1
🔎 Para tu informe

¿Qué número de VLAN tiene asignado el puerto eth1? ¿Qué VLANs transporta el puerto trunk eth5?

A6 Verificar aislamiento de VLANs 10 pts
🖥 Guacamole — Sesión requerida

Conectate a OVS-SW desde Guacamole.

Verificá que el switch filtra correctamente el tráfico entre VLANs a nivel de capa 2:

A7 Captura de tráfico con tcpdump → Wireshark 20 pts
🖥 Guacamole — Dos sesiones simultáneas

Abrí dos pestañas en tu navegador: una con sesión a OVS-SW y otra con sesión a PC1.

Paso 1 — En la sesión de OVS-SW: iniciá la captura en el puerto trunk hacia R1 (solo 20 paquetes ICMP):

En OVS-SW
tcpdump -i eth5 -w /tmp/vlan-cap.pcap icmp -c 20

Paso 2 — En la sesión de PC1: generá tráfico ICMP mientras tcpdump captura:

En PC1 (mientras tcpdump está corriendo en OVS)
ping -c 20 192.168.20.10   # Ping a PC2 (distinta VLAN — pasa por trunk)

Paso 3 — En OVS-SW: convertí el archivo pcap a base64 para poder copiarlo:

En OVS-SW
base64 /tmp/vlan-cap.pcap   # Imprime el archivo como texto base64

Seleccioná y copiá toda la salida base64 (usá Ctrl+Alt+Shift en Guacamole para acceder al portapapeles).

📦 Convertir base64 → pcap → abrir en Wireshark
  1. Pegá el texto base64 copiado en un archivo de texto en tu computadora y guardalo como vlan-cap.b64
  2. Windows (PowerShell):
    [System.Convert]::FromBase64String([System.IO.File]::ReadAllText("vlan-cap.b64")) | Set-Content -Encoding Byte "vlan-cap.pcap"
  3. Linux / Mac (terminal):
    base64 -d vlan-cap.b64 > vlan-cap.pcap
  4. Abrí Wireshark → File → Open → seleccioná vlan-cap.pcap
  5. En Wireshark, filtrá con icmp y expandí un paquete → buscá el campo "802.1Q Virtual LAN" → anotá el VLAN ID que aparece

09 — Conclusiones
¿Qué aprendiste hoy?

🐳 De los simuladores a la red real

Hoy trabajaste con software que se usa en producción real: Open vSwitch corre en los data centers de Amazon y Google; FRRouting es el motor de enrutamiento de muchos ISPs; tcpdump es la herramienta número uno de cualquier ingeniero de red cuando hay un incidente.

La diferencia entre un simulador como Packet Tracer y ContainerLab no es solo técnica — es conceptual. En PT modelás comportamientos. En ContainerLab ejecutás software real sobre una red virtual que se comporta exactamente como una red física.

Las VLANs que configuraste hoy, el trunk 802.1Q que capturaste con tcpdump, el router-on-a-stick que viste en FRRouting — todo eso existe, con los mismos comandos y los mismos protocolos, en la red que te conecta a Internet ahora mismo.
