Laboratorio 3: Infraestructura Corporativa con DNS y Simulación de Internet

Este laboratorio representa la consolidación de las arquitecturas previas. Romperemos el aislamiento de Capa 2 integrando un Router central que actuará como Gateway de las VLANs. Además, estructuraremos un segmento de DMZ/Internet pública para alojar un Servidor DNS e introduciremos enrutamiento estático hacia la WAN para que las PCs internas resuelvan dominios web reales simulados.

🌐 La Importancia del DNS en el DHCP: Al tener salida a redes externas, ahora es mandatorio que el Servidor DHCP (Server-DHCP) no solo entregue IP y Máscara a la VLAN 30, sino que también propague de forma automática la IP del Servidor DNS para que los hosts puedan navegar usando nombres de dominio (ej: www.cisco.com) en lugar de recordar IPs complejas.

Fase 1: Diseño de la Topología y Mapa de Direccionamiento

Evolucionaremos el escenario previo interconectando los switches de acceso a las interfaces del Router y creando un nuevo dominio para representar la nube/Internet.

Fase 2: Guía de Configuración de los Servicios Core

Ajustaremos el Servidor DHCP local para que distribuya el nuevo parámetro DNS, configuraremos las puertas de enlace en el Router y daremos de alta los registros de nombres.

Servicios Actualización del Server-DHCP (VLAN 30)
  1. Hacé clic en el Server-DHCP de la VLAN 30 e ingresá a Services > DHCP.
  2. En el campo Gateway colocá la IP del Router: 192.168.30.1.
  3. En el campo DNS Server colocá la IP de nuestra futura WAN: 8.8.8.8.
  4. Hacé clic en Save.
  5. Andá a la pestaña Desktop > IP Configuration del servidor y asegurate de agregarle su Gateway manual: 192.168.30.1.
Servicios Configuración del Servidor DNS Público (Internet)

Paso A: Asignación de la IP Pública

Hacé clic en el nuevo Server-DNS, ingresá a Desktop > IP Configuration y configurá de forma estática:

  • IP Address: 8.8.8.8
  • Subnet Mask: 255.0.0.0
  • Default Gateway: 8.0.0.1 *(IP de la interfaz Gi0/0/3 de R1)*

Paso B: Alta de Registros de Nombres (A Records)

  1. Andá a la solapa Services del servidor y seleccioná DNS.
  2. Cambiá el estado del servicio a ON.
  3. Crearemos un registro para simular una página web en el servidor DHCP (VLAN 30):
    • Name: www.telecom.lab
    • Type: A Record
    • Address: 192.168.30.5 *(La IP de nuestro Server-DHCP, que también tiene HTTP activo por defecto)*
  4. Hacé clic en Add (Añadir).
CLI Configuración de Interfaces y Gateways en R1

Entrá a la consola CLI del Router R1 para encender las interfaces físicas, asignarles sus respectivas IPs de Gateway por subred y habilitar el enrutamiento directo.

Router> enable
Router# configure terminal
Router(config)# hostname R1

# Interfaz hacia VLAN 10 (Ingeniería)
R1(config)# interface gigabitEthernet 0/0/0
R1(config-if)# ip address 192.168.10.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit

# Interfaz hacia VLAN 20 (Administración)
R1(config)# interface gigabitEthernet 0/0/1
R1(config-if)# ip address 192.168.20.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit

# Interfaz hacia VLAN 30 (Invitados y Servidor Local)
R1(config)# interface gigabitEthernet 0/0/2
R1(config-if)# ip address 192.168.30.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit

# Interfaz hacia la WAN / Red del Servidor DNS
R1(config)# interface gigabitEthernet 0/0/3
R1(config-if)# ip address 8.0.0.1 255.0.0.0
R1(config-if)# no shutdown
R1(config-if)# exit

R1(config)# do write memory

Fase 3: El Concepto de Redes Directamente Conectadas

Una duda común en este punto es: ¿Por qué no escribimos ningún comando de enrutamiento estático (como ip route) y aun así las computadoras se comunican entre distintas redes?

Teoría CCNA Enrutamiento Implícito de Capa 3

Los routers construyen su plano de reenvío basándose en diferentes fuentes. En este laboratorio, el éxito del tráfico se debe al concepto de Redes Directamente Conectadas (Directly Connected Networks).

  • Inyección Automática: Al asignar una dirección IP válida a una interfaz física (`ip address ...`) y levantarla eléctricamente (`no shutdown`), el sistema operativo Cisco IOS calcula automáticamente la red a la que pertenece y la inserta de forma inmediata en su tabla de enrutamiento global.
  • Rol de Nodo Central: Debido a que el Router R1 tiene un puerto físico (o "pie") conectado directamente en cada una de las cuatro subredes del mapa, él conoce de manera nativa los cuatro caminos. No requiere que un administrador le indique estáticamente cómo llegar a redes que ya están unidas a sus propios chasis.

Auditoría de la Tabla de Enrutamiento

Para comprobar esto empíricamente desde el plano de control, ejecutá el siguiente comando de diagnóstico en el modo privilegiado del Router (R1#):

R1# show ip route

Verás una salida similar a la siguiente en la sección inferior de la CLI. Prestá atención a las letras iniciales (códigos de origen):

Codes: C - connected, L - local, S - static, R - RIP, M - mobile ...

      8.0.0.0/8 is variably subnetted, 2 subnets, 2 masks
C        8.0.0.0/8 is directly connected, GigabitEthernet0/0/3
L        8.0.0.1/32 is directly connected, GigabitEthernet0/0/3
      192.168.10.0/24 is variably subnetted, 2 subnets, 2 masks
C        192.168.10.0/24 is directly connected, GigabitEthernet0/0/0
L        192.168.10.1/32 is directly connected, GigabitEthernet0/0/0
      192.168.20.0/24 is variably subnetted, 2 subnets, 2 masks
C        192.168.20.0/24 is directly connected, GigabitEthernet0/0/1
L        192.168.20.1/32 is directly connected, GigabitEthernet0/0/1
      192.168.30.0/24 is variably subnetted, 2 subnets, 2 masks
C        192.168.30.0/24 is directly connected, GigabitEthernet0/0/2
L        192.168.30.1/32 is directly connected, GigabitEthernet0/0/2
📋 Interpretación de Códigos de Examen:
• El prefijo C (Connected) identifica el rango de red general que el router puede enrutar de forma directa e inalámbrica hacia esa interfaz.
• El prefijo L (Local) (introducido a partir de la versión 15 de Cisco IOS) representa la IP específica de /32 asignada a la propia interfaz del hardware del router, optimizando el procesamiento de paquetes destinados al propio equipo.

Fase 4: Pruebas de Verificación y Control de Navegación Global

Con el Router operando en la Capa 3, realizaremos las pruebas lógicas de resolución de nombres y conectividad extremo a extremo.

Troubleshooting Validación Completa del Sistema

Prueba 1: Comprobación de la Renovación DHCP con Parámetro DNS

Andá a la PC-V30, abrí el Command Prompt (CMD) y forzá la actualización de la concesión IP:

C:\> ipconfig /release
C:\> ipconfig /renew
C:\> ipconfig

Verificá que ahora figure de forma correcta: Gateway 192.168.30.1 y DNS Server 8.8.8.8.

Prueba 2: Comprobar el Enrutamiento Inter-VLAN (Pings Exitosos)

A diferencia del Laboratorio 2, ahora el Router unifica las redes. Desde la PC-V10 (VLAN 10), tirale un ping a la PC de la otra punta del edificio y al servidor DNS externo:

C:\> ping 192.168.20.10   # Ping hacia PC-V20 (Exitoso)
C:\> ping 8.8.8.8         # Ping hacia el Servidor DNS en la WAN (Exitoso)

Prueba 3: Prueba de Fuego - Navegación por Nombre de Dominio (DNS)

  1. Hacé clic en cualquier PC (por ejemplo, PC-V30).
  2. Ingresá a la pestaña Desktop y seleccioná la aplicación Web Browser (Navegador Web).
  3. En la barra de direcciones URL, escribí de forma exacta: http://www.telecom.lab y presioná Go o Enter.
  4. Si todo está bien configurado, la PC le preguntará al servidor 8.8.8.8 qué IP tiene ese nombre, este le responderá que es la 192.168.30.5, y el navegador abrirá con éxito la página web por defecto de Cisco Packet Tracer alojada en el Servidor DHCP de la VLAN 30.
